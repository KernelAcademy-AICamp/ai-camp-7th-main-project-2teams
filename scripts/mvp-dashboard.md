# MVP Dashboard — Input·Output·Owner 실행 보드

**용도**: 출시 직후 그로스 국면에서 "무엇을 실행했고(Input) → 지표가 어떻게 움직였고(Output) → 누가 책임지는가(Owner)"를 한 화면에 잇는 **내부 PM 실행 보드**.
**성격**: 사용자향 A64(개인 대시보드)와 무관. 팀 내부 운영 도구.
**출처**: OKR은 `prd.md §4`, 계측 부채는 `business-viability.md` 기준.

> 이 보드 자체도 MVP다. 1단계는 노션/시트에 아래 표를 그대로 붙여 **수기 입력**부터 시작하고, 리추얼이 정착하면 Output 컬럼만 API 로그 기반으로 자동화한다.

---

## 컬럼 정의 (3컬럼)

| 컬럼 | 무엇을 적나 | 규칙 |
|---|---|---|
| **Input (실행)** | 이번 주 팀이 실제로 한 그로스 액션 1개 | 태스크 ID(A-id) 또는 실험명. "바쁨"이 아니라 "검증 가능한 베팅"만. |
| **Output (지표 변화)** | 그 Input에 연결된 **단 하나의** OKR 지표와 before→after | 숫자 필수. baseline 없으면 "측정 시작"이라 명시. |
| **Owner (담당자)** | 그 행을 끝까지 책임지는 사람 1명 | 반드시 1명 (공동 책임 금지). PRD §2 담당자 TBD를 여기서 실명으로 채운다. |

**행 추가 원칙**: 한 행 = 하나의 Input = 하나의 Output = 한 명의 Owner. 다대다 금지.

## Input 후보 백로그 (OKR별 — 매주 여기서 Input을 꺼내 위 보드로 올린다)

각 실행은 **연결될 Output(지표)**가 정해진 것만 백로그에 남긴다. 지표에 안 붙는 아이디어는 올리지 않는다.

### O2 · 초기 사용자 확보 (MAU 500 / 첫 저장 70% / 7일 리텐션 30%)
- [ ] A25 웹스토어 심사 통과 후 등록 페이지 A/B → **MAU**  · MKT
- [ ] 개발자 커뮤니티 3곳 게시 (research.md §3 얼리어답터) → **MAU** · MKT
- [ ] A26 온보딩 첫 저장까지 스텝 축소 → **첫 저장 완료율** · FE
- [ ] A29/A30 파일 임포트로 초기 저장량 부트스트랩 유도 → **첫 저장 완료율** · FE
- [ ] 7일차 리마인드 메일 / 재방문 트리거 → **7일 리텐션** · PM

## 운영 리추얼 (주 1회, 15분)

1. **월요일**: 지난주 각 행의 Output 숫자 채우기 → "먹혔나/안 먹혔나" 판정.
2. 2주 연속 flat인 Input은 **Owner가 손절 사유 한 줄** 남기고 백로그로 내림.
3. 백로그에서 이번 주 Input 최대 5개 선정 (지표에 붙는 것만).
4. baseline 없는 지표는 이번 주 **"측정 시작"**으로라도 반드시 한 행 배정 → `business-viability.md`의 "실측 없음" 부채 상환.

## 2단계 자동화 메모 (리추얼 정착 후)

Output 컬럼만 읽기 전용 집계로 자동화한다. 신규 저장 없음:
- **MAU / 첫 저장 완료율 / 1인당 저장 건수**: A5 저장 로그 + `withAuth` 인증 이벤트 집계
- **검색 사용률**: A7 `/api/search` 호출 유저 수 / MAU
- **응답시간**: A5 파이프라인 타이밍 로그 p95
- 산출물: 내부 `/admin/metrics` 읽기 전용 1페이지
- **보안 준수**: `embedding`·`content` 절대 미노출, `select('*')` 금지 — `.claude/rules/security.md` 그대로.
